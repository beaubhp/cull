# Cull Benchmark

This benchmark compares Cull with the two closest Python dead-code baselines:
Vulture and deadcode. It uses one fixed set of artificial but realistic Python
projects with exact expected findings.

The projects are artificial because precision and recall require ground truth.
Many real repositories contain little confirmed dead code, and full manual
recall labeling of large real projects is expensive and subjective. These
projects are instead designed to look like code developers commonly inherit:
multi-module services, libraries, CLIs, pipelines, workers, plugin systems,
configuration packages, and utility-heavy AI-era codebases.

The benchmark measures:

- true positives;
- false positives;
- false negatives;
- precision;
- recall;
- F1;
- median wall-clock runtime;
- peak RSS where the platform exposes it.

It does not measure auto-fix safety, runtime coverage, dependency cleanup,
commented-out code, or every possible Python lint rule.

## Tool Scope

The required tools are:

| Tool | Why included |
| --- | --- |
| Cull | Subject under evaluation. |
| Vulture | Classic Python dead-code detector. |
| deadcode | Newer whole-codebase Python unused-code detector. |

Ruff, Pylint, Pyflakes, Flake8, autoflake, pycln, and unimport are not included
in the headline comparison. They overlap on some unused-import or unused-local
checks, but they are linters or cleanup tools rather than direct whole-project
dead-code analyzers.

Skylos is a plausible future adapter because it also markets Python dead-code
detection. It is not required for this first benchmark; adding it should not
change the expected findings.

## Corpus

The checked-in corpus contains 15 artificial projects under `projects/` and one
expected-finding file per project under `expected/`.

Targets:

- 15 projects;
- 45k-80k total Python LOC;
- 250-500 Python files;
- 500-900 expected findings;
- two clean projects with zero expected findings;
- at least two large or noisy projects.

The expected finding categories are:

- `unused_import`;
- `unused_local`;
- `unreachable_statement`;
- `unused_function`;
- `unused_class`;
- `unused_private_method`.

## Expected Findings

Each expected file has this shape:

```json
{
  "project": "api_taskforge",
  "root": "projects/api_taskforge",
  "expected": [
    {
      "id": "api_taskforge:unused_import:001",
      "category": "unused_import",
      "path": "src/api_taskforge/routes/reports.py",
      "name": "json",
      "line": 8,
      "column": 1,
      "end_line": 8,
      "rationale": "Imported alias is never read by a may-execute reference."
    }
  ]
}
```

The expected files are the ground truth. They are not generated by the runner
and should not be edited to hide inconvenient tool output.

## Tool Output Mapping

The runner maps each tool's native output into the benchmark categories before
scoring:

| Benchmark category | Cull | Vulture | deadcode |
| --- | --- | --- | --- |
| `unused_import` | `CULL005` import bindings | unused `import` | `Import` |
| `unused_local` | `CULL006` local bindings | unused `variable` | `Variable` / `Name` |
| `unreachable_statement` | `CULL007` statement ranges | unreachable code | unreachable diagnostics |
| `unused_function` | `CULL001` / `CULL003` functions | unused function or public method | `Function` or public `Method` |
| `unused_class` | `CULL002` / `CULL004` classes | unused class | `Class` |
| `unused_private_method` | `CULL008` private methods | unused private method | private `Method` |

Parsed lines outside these comparable categories are surfaced as runner
warnings rather than silently scored.

## Scoring

Every expected finding not matched by a tool is a false negative.

Every parsed tool finding in a scoreable category that does not match an
expected finding is a false positive. This includes findings in clean projects.

Matching is deterministic:

- imports match by category, path, name, and import line;
- local bindings match by category, path, name, and binding line;
- functions, classes, and private methods match by category, path, name, and
  definition line;
- unreachable statements match by category, path, and overlap with the expected
  unreachable range;
- duplicate reports count once as a true positive and then as false positives.

Cull's headline score uses high-confidence findings. The JSON and Markdown
reports also include a clearly marked Cull high-plus-review aggregate.

## Runtime

Runtime includes subprocess startup time. That reflects command-line user
experience and avoids special-casing tools written in different languages.

The default policy is:

- one warmup run per tool/project;
- five measured runs per tool/project;
- median measured wall time in the report;
- same project path and machine for each tool;
- no network access needed during benchmark execution after tools are available;
- command lines, tool versions, Python version, OS, CPU, memory, and Cull commit
  recorded in the result JSON.

Peak RSS is recorded on platforms where `/usr/bin/time -l` exposes it. If RSS is
unavailable, the result uses `null`.

## Running

Prerequisites:

- Python 3;
- a release Cull binary when running Cull;
- `uvx` when running the Vulture or deadcode baselines.

The runner pins Vulture and deadcode in the command line. The first baseline run
may need network access to fetch those pinned tools; after that, the benchmark
itself does not download or generate the corpus.

Validate the corpus and expected files:

```bash
python3 benchmark/run.py --validate-only
```

Run runner self-tests:

```bash
python3 benchmark/test_run.py
```

Build Cull and run the full benchmark:

```bash
cargo build --release
python3 benchmark/run.py --cull target/release/cull --tools cull,vulture,deadcode --runs 5 --results benchmark/results/latest.json
```

During development, run a subset:

```bash
python3 benchmark/run.py --cull target/release/cull --tools cull --runs 1 --project api_taskforge
```

Generated results are written under `benchmark/results/` and ignored by default.
Raw tool outputs are retained under `benchmark/results/raw/`.

## Interpreting Results

The Markdown report contains:

1. aggregate summary table;
2. per-project table;
3. per-category table;
4. tool and environment metadata;
5. false-positive findings;
6. false-negative findings.

The benchmark supports a conservative claim about this corpus only. It should
not be read as proof that any tool is universally better on every Python
project.
